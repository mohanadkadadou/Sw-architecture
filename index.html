Software Architecture & Architect Roles
الباب الأول — تعريف المعمارية وأدوار المهندس المعماري
🏛️ What is Software Architecture?

Definition (Booch 1999): "Software architecture encompasses the set of significant decisions about the organization of a software system — selection of structural elements and their interfaces, behavior as specified in collaborations among those elements, and composition into larger subsystems."

  • In early literature: called "Programming in the Large"
  • Presents a view of a system as components + connectors
  • A body of methods and techniques to manage software complexity
  • Natural extension of the software engineering discipline
👨‍💼 The 4 Roles of a Software Architect

🔧 What is CBSE?

Component-Based Software Engineering (CBSE) is an approach to software development that relies on software reuse. It emerged from the failure of object-oriented development to support effective reuse.

📋 CBSE Essentials (4 Points)
  • 1. Independent components specified by their interfaces
  • 2. Component standards to facilitate component integration
  • 3. Middleware that provides support for component inter-operability
  • 4. A development process geared to reuse
🎯 CBD Key Goals & Problems
✅ CBD Goals (3)
1. Save time & money on large systems
2. Enhance software quality (component quality is key)
3. Detect defects within systems (challenging in CBD)
⚠️ CBD Problems (4)
1. Trustworthiness — no source code available
2. Certification — who certifies quality?
3. Emergent property prediction — how do compositions behave?
4. Requirements trade-offs — between competing components
🔌 Component Models & Composition

Component Models define standards for implementation, documentation, and deployment:

  • EJB — Enterprise Java Beans
  • COM+ — .NET Model
  • CORBA — Common Object Request Broker Architecture
  • NodeJS

Component Composition = assembling components to create a system. Requires writing "glue code" to integrate components.

Design Patterns in Software Engineering
الباب السابع — أنماط التصميم (سؤال الاختبار Q4 مباشرة)
⭐ Exam Q4 (13 pts): "What is a design pattern in software engineering, and WHY is it useful?"
📖 What is a Design Pattern?

A design pattern is a general, reusable solution to a commonly occurring problem within a given context in software design.

  • NOT a finished design — it's a template or blueprint
  • Describes the problem, the solution, when to apply it, and its consequences
  • Originally described by the "Gang of Four" (GoF) — Gamma, Helm, Johnson, Vlissides (1994)
  • 23 classic patterns organized into 3 categories
🎯 WHY are Design Patterns Useful?
  • Proven solutions — battle-tested by experienced developers
  • Common vocabulary — "Let's use Observer here" instantly understood by all developers
  • Avoid reinventing the wheel — reuse known working solutions
  • Improve code quality — maintainability, flexibility, extensibility
  • Easier communication — patterns provide a shared language between architects and developers
  • Reduce technical debt — proper structure from the start

• Composite — tree structure of objects

💡 Key Patterns with Examples
Singleton (Creational)
Creational
Ensures a class has ONLY ONE instance
Provides global access point to that instance
Example: Database connection, Logger, Config manager
Adapter (Structural)
Structural
Converts interface of a class into another interface clients expect
Makes incompatible interfaces work together
Example: Plugging European device into US socket (adapter)
Observer (Behavioral)
Behavioral
Defines one-to-many dependency between objects
When one object changes state, all dependents notified
Example: Event listeners, MVC (Model notifies Views)
Factory Method (Creational)
Creational
Defines interface for creating an object
Subclasses decide which class to instantiate
Example: UI Button creation on different platforms
Facade (Structural)
Structural
Provides simplified interface to complex subsystem
Hides complexity from client code
Example: Home theater system — one remote for all devices
Strategy (Behavioral)
Behavioral
Defines family of algorithms, encapsulates each one
Makes algorithms interchangeable at runtime
Example: Sorting algorithm selection, payment methods

1. URI Resource Identification — resources identified by URIs (e.g., /api/users/123)

3. Self-Descriptive Messages — content in many formats: JSON, XML, HTML, PDF
4. Stateless Interactions — requests are self-contained; no session state stored on server

Statelessness: 'The server does NOT need to know anything about what state the client is in. Both server and client can understand any message received, even without seeing previous messages.'

REST vs SOAP:
• REST is FASTER (no strict specification), consumes fewer resources
• REST is SIMPLER (uses standard HTTP methods)
• REST is MORE FLEXIBLE (JSON/XML/HTML — SOAP is XML-only)
• REST is MOBILE-FRIENDLY (lightweight JSON)
• REST is COMPATIBLE with SOAP (can use as middleware)
REST = Representational State Transfer. 4 مبادئ: 1.URI لتعريف الموارد 2.واجهة موحدة (GET/POST/PUT/DELETE) 3.رسائل وصفية (JSON/XML/HTML) 4.stateless interactions. Stateless: الخادم لا يحفظ حالة العميل. REST أفضل من SOAP: أسرع+أبسط+أمرن+mobile-friendly.

Client-Server Separation in REST:

Give a complete example of applying the Observer pattern. How does it relate to MVC?

Observer Pattern:
'Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.'

Components:
• Subject (Observable): The object being observed. Maintains list of observers. Notifies them on change.
• Observers: Objects that want to be notified of changes

Complete Example — Stock Price System:
Subject: StockMarket (price changes)
Observers: MobileApp, WebDashboard, EmailAlert, SMS Service
When AAPL price changes → Subject calls update() on all registered observers → each observer shows new price in their own way

Relation to MVC:
• MODEL is the Subject
• VIEWs are the Observers
• When Model data changes, it NOTIFIES all registered Views
• Each View updates its display independently
• This is the CORE mechanism enabling MVC's loose coupling
• Controller can add/remove Views as observers dynamically
Observer: علاقة واحد-لكثيرين. Subject (المُراقَب): يحتفظ بقائمة Observers ويُخطرهم عند التغيير. مثال: StockMarket (Subject) → MobileApp+WebDashboard+EmailAlert (Observers). ارتباطه بـ MVC: Model=Subject، Views=Observers — عندما يتغير Model يُخطر جميع Views تلقائياً.
